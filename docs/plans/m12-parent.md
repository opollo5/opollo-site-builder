# M12 — Brief-driven Sequential Page Generation

## What it is

A new generation mode that accepts a single whole-site brief (text or Markdown today; PDF/.docx stretch) and produces the full site one page at a time, each page generated with the entire brief in context plus a running summary of what preceding pages said. The operator uploads the document, a structural parser extracts an ordered per-page list with a per-page mode (`full_text` when the brief contains a complete section for that page, `short_brief` when it only sketches one), the operator reviews and commits the parse, and a sequential runner walks the list: draft → self-critique → revise → visual-review → revise again, pausing after each page for operator approval before advancing. DS-only generation is the default — no locked template per page, the runner composes from the site's design system and a shared `site_conventions` document that is fixed at page 1 and re-used verbatim for pages 2..N. Brand voice and design direction are first-class fields on the brief record, not hidden inside the document text, so the operator can edit them between pages without re-uploading. Pause-mode is the only review mode at launch; auto-advance is deferred until we've watched a dozen real sites go through the loop.

## Why a separate milestone

The existing batch generator (M3) and regeneration worker (M9) are per-page workers with no cross-page state — each slot is a leaf job, fan-out parallel, no ordering guarantees. Brief-driven generation inverts every one of those assumptions: it is sequential by contract (page 2 depends on page 1's committed conventions), it carries a growing content-summary context forward, it has a per-page review checkpoint the operator must clear, and it wants a multi-pass quality loop (self-critique + visual critique) that's too expensive to run on a 60-slot parallel batch. Folding this into M3 would either force the batch worker to learn a second execution shape or would quietly degrade batch throughput. M12 gets its own tables (`site_briefs`, `brief_pages`, `brief_runs`), its own runner (`lib/brief-runner.ts`), its own admin surface (`/admin/sites/[id]/briefs/[briefId]`), and reuses M7's publish path + M6's DS rendering without touching them.

The milestone is also write-safety-critical on a different axis than M3: a single brief run can spend 10-20× the Anthropic tokens of a batch slot (multi-pass × N pages × whole-doc context), so budget enforcement, idempotency, and resume-after-pause all have to land before the runner can touch a production tenant.

## Scope (shipped in M12)

- **M12-1 — schema + upload + structural parser + operator commit.** New tables `site_briefs`, `brief_pages`, `brief_runs` with the soft-delete + audit-column contract from `docs/DATA_CONVENTIONS.md`. Upload surface at `/admin/sites/[id]/briefs/new` accepts `text/plain` and `text/markdown`; file is stored in Supabase Storage under `site-briefs/<site_id>/<brief_id>.md`. Structural parser (`lib/brief-parser.ts`) prefers explicit structure first — Markdown H1/H2 sections, `---` separators, numbered "Page 1:" / "Page 2:" headers — and falls back to a single Claude inference call when no structure is detectable. Parser output is a draft `brief_pages` list with per-page `mode in ('full_text','short_brief')` based on whether the extracted section is long enough (>= 400 words) to be a complete page spec. Operator reviews the parse at `/admin/sites/[id]/briefs/[briefId]`, can re-order rows, edit titles, flip modes, add/remove pages, then commits — commit freezes the list and unlocks the runner.
- **M12-2 — brand voice + design direction + `site_conventions` schema + anchor-page spec.** `site_briefs` gains `brand_voice text` and `design_direction text` first-class fields populated from a form (pre-filled with Claude-inferred defaults the operator can override). New `site_conventions` JSONB column on `site_briefs` holds the exact design+content conventions the runner will reuse verbatim across every page: typographic scale choice, section rhythm, tone register, CTA phrasing, hero pattern, color-role mapping over the DS palette. The anchor-page spec (applied to page 1) runs 2-3 extra revision cycles beyond the standard multi-pass — the runner promotes the anchor's final `site_conventions` choices into the brief row, and pages 2..N receive them as hard constraints rather than suggestions.
- **M12-3 — sequential runner with multi-pass, whole-doc context, quality gates.** `lib/brief-runner.ts` processes one `brief_pages` row at a time. Per-page loop: draft (whole brief + `site_conventions` + running content summary + page spec) → self-critique (Claude re-reads its own draft against the brief and conventions, returns a list of gaps) → revise (applies the critique). Page 1 runs the loop 2-3 extra times to stabilise `site_conventions`. Output is staged in `brief_pages.draft_html` until the operator approves; approval promotes the draft to `generated_html` and appends a compressed 2-3 sentence summary to the brief's running `content_summary` that pages N+1.. will read. Quality gates reuse M7's runner contract (HTML size cap, DS-token whitelist, link integrity) — a gate failure blocks approval until the runner re-revises.
- **M12-4 — visual review pass.** After the multi-pass text loop lands a candidate, the runner spins up a Playwright worker to render `draft_html` against the site's DS, captures a viewport + full-page screenshot, and feeds both to Claude with the original page spec + `site_conventions`. Claude returns a structured critique (layout issues, visual hierarchy, contrast, white-space density, CTA prominence). If the critique flags any severity-high issue, the runner applies a revision pass that includes the screenshot feedback and re-renders. Capped at 2 visual iterations per page to bound cost. The final screenshot + critique log are stored against the `brief_pages` row so the operator sees exactly what Claude saw when approving.
- **M12-5 — review-between-pages UI (approve / cancel / resume).** `/admin/sites/[id]/briefs/[briefId]/run` is the runner surface. Shows the committed page list with per-row status (pending / generating / awaiting_review / approved / failed / cancelled), the current page's draft HTML preview + screenshot + Claude critique, and three action buttons: **approve** (advances the runner to the next page), **revise with note** (appends an operator note to the next revision pass), **cancel** (halts the run, leaves generated pages in place so the operator can keep or discard per page). The runner is pause-only: on every page completion it transitions to `awaiting_review` and waits for the operator action. Auto-advance is explicitly out of scope for M12.
- **M12-6 — E2E + docs + PDF/.docx parser (stretch).** Playwright spec covering upload → parse review → commit → run → approve page 1 → approve page 2 → cancel at page 3 → publish approved pages via the existing M7 surface. New `docs/patterns/brief-driven-generation.md` capturing the runner contract, the `site_conventions` anchor protocol, and the per-page review state machine. Stretch: if budget permits, add a PDF/.docx parser variant (`pdf-parse` + `mammoth`) gated behind a feature flag; if it slips, a BACKLOG entry carries it to a follow-up slice without blocking M12-5's merge.

## Out of scope (tracked in BACKLOG.md)

- **Auto-nav / menu generation.** The runner produces page HTML; it does not mutate the site's WP menu. Operator wires menus by hand after publish. Likely lands in a dedicated "site chrome" slice that also covers footer, header nav, breadcrumb generation — too large to bolt onto M12.
- **`sitemap.xml` generation.** Same reasoning. WordPress plugins already produce sitemaps on publish; we do not want to own that surface until we have a reason to.
- **Multi-language briefs.** One brief = one language today. A multi-language brief would need per-language `site_conventions`, per-language running summaries, and per-language review surfaces — a 2-3× scope multiplier. Belongs to an i18n milestone.
- **Brief versioning + diff.** Today an edited brief is a new brief row (soft-delete the old one). Full version history + per-field diff + "re-run page 4 with v2 of the brief" is a separate capability that would need its own data model. Not needed for the first dozen real sites.
- **Auto-advance review mode.** Pause-only for launch. Auto-advance is a one-flag change once we trust the visual-review pass — but "trust" here means watching enough real runs that we know what a bad visual critique looks like. Deferred explicitly.

## Env vars required

None new beyond existing `ANTHROPIC_API_KEY`, the Supabase trio (`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`), and the already-provisioned Langfuse + Playwright setup M7/M11 rely on. The visual-review pass reuses the existing Playwright worker infrastructure; no new screenshot service, no new storage bucket beyond the existing `site-briefs/` prefix added in M12-1.

## Write-safety + design-quality contract

### Sequential runner concurrency (M12-3)

The runner is a `concurrency=1` worker keyed on `brief_id`. Only one pass on one page of one brief can be in flight at a time. The lease + heartbeat shape is reused verbatim from the M3 batch worker: a row in `brief_runs` with `worker_id`, `lease_until`, and `heartbeat_at` is claimed via `UPDATE ... WHERE status='queued' AND (lease_until IS NULL OR lease_until < now())` and refreshed every 30s. A DB unique partial index `ON brief_runs (brief_id) WHERE status='running'` enforces the "one active run per brief" invariant at the database level — two workers racing the same brief get a constraint violation on the second, not a corrupted shared state. Cross-brief parallelism is allowed: different briefs run on different workers concurrently.

### Multi-pass per page (M12-3)

Each page runs through up to three text passes — draft, self-critique, revise — and up to two visual passes. Each pass writes to `brief_pages.draft_html` under an idempotency key of `(brief_id, page_ordinal, pass_kind, pass_number)`. A retry of an interrupted pass reads the key, sees the row already exists, and short-circuits rather than re-billing Anthropic. Pass output is never the final authoritative state — the operator's approve action is what promotes `draft_html` to `generated_html`. Between passes we store the critique output separately (`brief_pages.critique_log` JSONB) so a re-run can use the same critique without re-calling Claude.

### Visual review pass (M12-4)

The Playwright renderer runs in a hermetic sandbox: no network egress beyond the site's DS CDN, no cookies, no service-role headers. Screenshots are written to an ephemeral worker tmpdir, base64-inlined into the Claude multi-modal call, and discarded as soon as the critique returns. **Screenshots are never persisted to Supabase Storage and never logged** — the critique text is what lands on `brief_pages.critique_log`. If an operator needs to re-see what Claude saw, the approve surface re-renders from the current `draft_html` on demand. This retention policy is enforced by the worker contract, not by a retention job — there's nothing to retain.

### Cost controls (M12-4 amendment — 2026-04-24)

The visual-review pass is the first place a single page can run 5+ billed Claude calls (3 text + 2 visual critiques + any visual-revise text passes). "Capped at 2 visual iterations" is not sufficient on its own — an operator who chose Opus for both text and visual tiers on a 40-page site can still rack up multi-hundred-dollar runs if every page maxes its iteration budget. M12-4 ships five cost-control invariants that land in the runner, data model, and pre-flight path:

1. **Hard revision cap + best-effort commit.** Per-page visual iterations cap at 2. If the critique after iteration 2 still returns severity-high issues, the runner commits the current `draft_html` with `brief_pages.quality_flag = 'capped_with_issues'` and proceeds to `awaiting_review`. No infinite loops. Operator sees the flag in the review surface (M12-5 renders it) and decides whether to accept, edit, or cancel. Invariant: **the runner never runs a third visual iteration on the same page, regardless of critique severity.**

2. **Per-page combined-cost ceiling (default 200 cents).** `brief_pages.page_cost_cents` accumulates the cost of every Anthropic call against the page (text draft, self-critique, revise, visual critique × N, visual revise × N). Before each visual iteration, the runner checks `page_cost_cents + projected_iteration_cost > BRIEF_PAGE_COST_CEILING_CENTS`. If yes, it stops the visual loop, sets `quality_flag = 'cost_ceiling'`, and transitions to `awaiting_review`. The $2 default is configurable per tenant via `tenant_cost_budgets.per_page_ceiling_cents_override` (NULL means use the default).

3. **Operator cost visibility.** `brief_pages.page_cost_cents` + `brief_runs.run_cost_cents` (rolled up on every pass completion) are returned by the admin read path that M12-5 wires into its page-list surface. Formatted as cents-to-dollars at the UI layer. The schema carries cents to avoid float drift.

4. **Model tier per brief + per pass-kind.** `briefs.text_model` and `briefs.visual_model` are first-class columns (default `claude-sonnet-4-6` and `claude-sonnet-4-6` respectively). The operator-facing brief-commit form (M12-5 surface) lets the operator override per-brief. `RUNNER_MODEL` in `lib/brief-runner.ts` is replaced by these per-brief fields — no hard-coded Opus. Guard: runner validates that `briefs.text_model` and `briefs.visual_model` are in the `ANTHROPIC_MODEL_ALLOWLIST` constant at pass-start time; an unknown model ID fails the page with `INVALID_MODEL` rather than sending an unbounded-cost call to whatever string is in the column.

5. **Pre-flight cost estimate + budget confirmation.** New helper `estimateBriefRunCost(briefId)` computes `N_pages × avg_cost_per_page(text_model, visual_model)` using the pricing table in `lib/anthropic-pricing.ts`. `lib/briefs.startBriefRun()` (the function M12-5's "start run" action calls) pulls the estimate + the tenant's remaining monthly budget from `tenant_cost_budgets` and returns it. If `estimate > 0.5 × remaining_budget`, the helper returns `{ ok: false, code: 'CONFIRMATION_REQUIRED', estimate_cents, remaining_budget_cents }`; the UI re-submits with `confirmed: true` to proceed. This is a soft gate (operator can override) — the hard gate is the existing `reserveWithCeiling` path which halts the run mid-flight if actuals exceed the monthly cap.

These invariants do not replace the existing per-page `reserveWithCeiling` path (which reserves against the tenant monthly cap up-front); they layer on top of it. `reserveWithCeiling` answers "can we afford the worst case?"; the per-page $2 ceiling answers "did this particular page run out of control?".

### Whole-document context (M12-3)

Each pass receives the full brief text as a `<brief_document>`-tagged input at the top of the user turn, followed by `<site_conventions>` (exact JSONB serialised), `<content_summary>` (the running compressed summary), and `<page_spec>` (the specific page's parsed section). Token budget per pass is capped at 60k input tokens; a brief exceeding that cap (a ~45k-word document) is rejected at upload time with `BRIEF_TOO_LARGE`. The whole-doc payload is cache-keyed via Anthropic prompt caching so passes 2..N on the same page reuse the brief + conventions segments without re-billing — critical to keeping multi-pass cost tractable.

### Site-conventions capture (M12-2 + M12-3)

`site_conventions` is written exactly once per brief — at the end of page 1's anchor cycle — and is read verbatim by pages 2..N. It is stored exact (structured JSONB, not prose) so pages 2..N receive hard constraints rather than a re-summarisation that could drift. The content summary, by contrast, is compressed on every append because it grows linearly with page count and would blow the token budget by page 20 if stored verbatim. The asymmetry is deliberate: conventions must not drift, content summary must not balloon.

### First-page anchor (M12-2 + M12-3)

Page 1 runs the standard multi-pass loop plus 2-3 additional revise cycles specifically to stabilise `site_conventions`. After the anchor cycles complete, the runner promotes the final conventions JSONB onto `site_briefs.site_conventions` (single-row update guarded by `version_lock`) and transitions page 1 to `awaiting_review`. Page 2 cannot start until the operator approves page 1 — the runner's page-ordinal state machine refuses to advance past page 1 while its status is not `approved`. This is the anchor gate: a bad voice or layout is caught at page 1 before N-1 downstream pages inherit it.

### Resume after crash (M12-3)

`brief_runs.current_ordinal` tracks the page index the runner is on; `brief_pages.current_pass_kind` and `current_pass_number` track the pass. On worker restart, the resume path reads the `brief_runs` row, sees `current_ordinal` + the corresponding `brief_pages` row's pass pointers, and hands control back to the runner at exactly that pass. The per-pass idempotency key ensures a pass that was mid-billing when the worker died is retried at-most-once without duplicating the write. Operator-visible state (`awaiting_review`, `approved`, `cancelled`) is never rolled back by a resume — resume only re-enters in-progress passes.

## Performance notes

- **Wall-clock per run.** A representative 30-40 page site, each page running draft + self-critique + revise + up to two visual revise cycles, plus the 2-3 anchor cycles on page 1, plus operator-review pauses, lands at roughly 1.5-2 hours of wall-clock from commit to publish. The runner spends most of that time waiting on Anthropic responses; the operator spends most of the pause time reading a single page at a time. This is an intentional tradeoff — M3 ships a parallel batch in 10-15 minutes; M12 takes 10× longer to produce substantively better output.
- **Per-page token cost.** Each page is 3-5× the Anthropic spend of a single-shot M3 slot: multi-pass (3 text passes × whole-doc context) + visual review (up to 2 multi-modal calls) + anchor extras on page 1. Prompt caching on the `<brief_document>` + `<site_conventions>` prefix keeps passes 2..N on the same page near-free on input tokens; output tokens still bill in full.
- **Running-summary budget.** The content summary is capped at ~2k tokens. On every page approval the runner appends a 2-3 sentence summary of the just-approved page; every 10 pages a Claude compact pass rewrites the whole summary tighter. This keeps the context window flat as the site grows instead of expanding linearly — the compact pass costs one extra Claude call every 10 pages, which is a negligible tax against the savings on pages 11-40's input budget.
- **Visual-review I/O.** Screenshots are PNG blobs in the 200-800KB range per page. They're base64-inlined into the Claude multi-modal call and discarded — no Storage write, no log line, no retention cost. The worker tmpdir is scoped per-page and unlinked on pass completion.
- **Cross-milestone dep: M8 budget guard.** Because per-page cost is variable (anchor adds 2-3 cycles, visual review adds 0-2 cycles, worst-case page is ~7× a single-shot M3 slot), the M8 budget reserve path needs to be capacity-aware — reserving the worst-case ceiling per page up-front and releasing the unused delta at page completion. Today's `reserveBudget` reserves a fixed estimate; M12-3 extends it with a `reserveWithCeiling()` variant. This is flagged as a cross-milestone dependency on M8 — not a blocker for M12-1/M12-2 but a hard prerequisite for M12-3's merge.

## Risks identified and mitigated

1. **Unbounded token growth past page 20.** → Running content summary is capped at ~2k tokens with a Claude compact pass every 10 pages. Input-token cost on page N is constant-bounded by `brief_size + site_conventions + 2k summary + page_spec`, not linear in N. Test: synthesise a 40-page brief fixture, assert pass-1 input-token count on page 40 is within 10% of page 10's.

2. **Parser hallucinates pages not in the document.** → Structural-first parse returns only sections the parser found in the source text, tagged with their byte offsets. The Claude-inference fallback uses `<brief_document>`-tagged input prompting and returns a JSON list with per-entry source-quote citations; any entry without a matching span in the source fails validation and is dropped. Regardless of parser path, the operator must commit the list before the runner starts — a hallucinated page that slips through still requires an operator click to land.

3. **Mid-run worker crash loses progress.** → `brief_runs.current_ordinal` + `brief_pages.current_pass_kind`/`current_pass_number` pointers identify the exact pass a restart should re-enter. Per-pass idempotency key `(brief_id, page_ordinal, pass_kind, pass_number)` prevents duplicate Anthropic billing on the retry. Test: kill the worker mid-pass 2 of a 3-page brief, restart, assert the run completes without a duplicate billing row.

4. **Operator edits `brief_pages` body mid-run.** → Every `brief_pages` row carries `version_lock`; the runner reads+writes under optimistic concurrency. An operator edit to a future page is accepted but only applies when the runner reaches that page — edits to already-generating or already-generated pages return VERSION_CONFLICT and are surfaced to the operator as "page N is in flight; cancel the run to edit it". Already-approved pages are read-only.

5. **Concurrent runs on the same brief.** → DB partial unique index `ON brief_runs (brief_id) WHERE status='running'` rejects a second claim at constraint-violation time. A second operator clicking "start run" while another run is active gets a clear error, not a corrupted shared state. Covered by the `concurrency-test-harness.md` two-workers-race-same-brief test.

6. **Billed parse call duplicated on retry.** → Idempotency key on the Claude-inference fallback is `hash(brief_id + sha256(source_text))`. A retry replays the same key and short-circuits to the cached response. Structural parse is deterministic and not billed, so its retry is free.

7. **PII in uploaded briefs.** → Supabase RLS scopes `site_briefs` + `brief_pages` + `brief_runs` to the owning site's tenant; no cross-tenant read is possible. Langfuse traces redact the brief body to a SHA256 digest + byte count before emitting; the actual text is never shipped off-platform. Operator-visible surfaces render from the primary DB, not from the trace store.

8. **Visual review screenshot leaks sensitive content.** → Screenshots live in the worker tmpdir for the duration of a single Claude multi-modal call, are base64-inlined into that call, and are unlinked before the pass returns. No Storage write. No log line that includes screenshot bytes. No retention job. The critique text that persists on `brief_pages.critique_log` is plain language — it describes layout, not copy.

9. **Quality degrades past page 20 as context compresses.** → Eval set lives under `lib/brief-runner/__evals__/` with a fixture 40-page brief + golden critique outputs at pages 1, 10, 20, 30, 40. CI runs the eval on every M12-3+ PR and fails if pass-2 self-critique on a late page misses a brief-required detail the early-page run caught. Pause-mode-only for launch gives a human review checkpoint per page regardless of what the eval catches.

10. **Multi-pass cost blowup.** → Hard cap of 3 text passes + 2 visual passes per page enforced in the runner loop (not in the DB — the runner refuses to start pass 4). Per-page ceiling `reserveWithCeiling()` checks against the tenant's monthly cap before the first pass fires; if the worst-case ceiling would exceed the remaining budget, the run halts with `BUDGET_EXCEEDED` before spending a token.

11. **DS-only generation produces incoherent layouts across pages.** → `site_conventions` is locked after page 1's anchor cycle and stored as exact JSONB, not prose. Pages 2..N receive it verbatim as hard constraints. The conventions schema includes typographic scale, section rhythm, hero pattern, and CTA phrasing — the four dimensions most responsible for the "this feels like one site" feel.

12. **First-page anchor fails to produce usable conventions.** → Runner's page-ordinal state machine refuses to advance past page 1 while page 1 is not `approved`. If the anchor exhausts its cycle budget without converging (3 text + 2 visual passes), the runner halts with `ANCHOR_FAILED` and the operator decides whether to edit the brand voice / design direction and retry, or abandon the run. Page 2 never starts against a bad anchor.

13. **Revision-loop overspend.** → Visual iterations capped at 2 in the runner (not the DB). Per-page combined-cost ceiling (default 200 cents, tenant-overridable) is checked before each visual iteration; if `page_cost_cents + projected_iteration_cost > ceiling`, the runner stops, sets `brief_pages.quality_flag = 'cost_ceiling'`, and proceeds to `awaiting_review`. A third visual iteration is unreachable even if critique after iteration 2 is still severity-high — runner commits with `quality_flag = 'capped_with_issues'` instead. Test: every critique stub returns severity-high; assert iteration counter stops at 2 and page lands in `awaiting_review` with the correct flag. Second test: synthesise a per-page cost just under the ceiling, assert the next iteration is skipped and the flag is set.

14. **Model-tier drift — silent Opus spend.** → `RUNNER_MODEL` is not hard-coded. `briefs.text_model` + `briefs.visual_model` are first-class columns read by the runner at pass-start. Default is `claude-sonnet-4-6` for both; operator overrides at commit time (M12-5 form). Guard: runner validates the stored value against an `ANTHROPIC_MODEL_ALLOWLIST` at pass-start; unknown value → page fails with `INVALID_MODEL` rather than firing the call. Test: update `briefs.text_model` to `not-a-model`, tick the runner, assert the page fails with `INVALID_MODEL` and zero Anthropic calls fired.

15. **Operator-blind overspend.** → `brief_pages.page_cost_cents` + `brief_runs.run_cost_cents` are stamped on every pass completion (single UPDATE inside the same CAS transaction as the critique_log write). `lib/briefs.startBriefRun()` computes a pre-flight estimate from `N_pages × avg_cost_per_page(text_model, visual_model)` via the existing `lib/anthropic-pricing.ts` table. If `estimate > 0.5 × remaining_budget`, startBriefRun returns `CONFIRMATION_REQUIRED` with the estimate + remaining budget; the UI re-submits with `confirmed: true` to proceed. Test: seed a tenant budget where remaining is 300 cents and a brief whose estimate is 200 cents; assert startBriefRun returns `CONFIRMATION_REQUIRED` without `confirmed`, and proceeds with it. Second test: after one pass completes, assert `brief_pages.page_cost_cents` > 0 and `brief_runs.run_cost_cents` matches the sum of per-page costs.

## Relationship to existing patterns

- **M12-1** follows `docs/patterns/new-api-route.md` for the upload route, `docs/patterns/new-migration.md` for the three new tables + storage-bucket policy, and `docs/patterns/new-admin-page.md` for the parse-review surface. The structural-first parser is a small new shape, but a single parser inside one slice isn't worth promoting to a pattern on its own.
- **M12-2** follows `docs/patterns/new-migration.md` for the `brand_voice` / `design_direction` / `site_conventions` column adds and `docs/patterns/rls-policy-test-matrix.md` for the tenant-scoping tests on the new columns. The anchor-promotion idempotency check is one-off; no new pattern.
- **M12-3** is a **new shape — promote to pattern if reused.** The multi-pass runner (draft → self-critique → revise with per-pass idempotency keys, whole-doc context with prompt caching, running-summary compaction, per-page quality gates) shares its lease/heartbeat mechanics with `docs/patterns/background-worker-with-write-safety.md` but the per-pass state-machine + context-management shape is genuinely new. If a future milestone (copy-editing agent, translation runner, anything else that needs multi-pass iterative Claude calls) asks for the same shape, promote `lib/brief-runner.ts` into `docs/patterns/multi-pass-runner.md`. Until then, the pattern lives as the single implementation.
- **M12-4** is a **new shape — promote to pattern if reused.** Playwright render → screenshot → multi-modal Claude critique → revise is a loop we have not built before. It borrows the Playwright-worker shape from M7's quality-gate runner but the critique-feedback-into-revise pass is new. Flag for promotion to `docs/patterns/visual-critique-loop.md` on first reuse (e.g. a future design-review milestone that runs visual critique against already-published pages).
- **M12-5** follows `docs/patterns/playwright-e2e-coverage.md` for `e2e/briefs.spec.ts` and `docs/patterns/new-admin-page.md` for the run surface. The operator state machine (pending → generating → awaiting_review → approved / cancelled) is admin-UI-sized and doesn't need its own pattern; VERSION_CONFLICT handling mirrors M8-5's existing shape.
- **M12-6** follows `docs/patterns/playwright-e2e-coverage.md` for the full-loop E2E + `docs/patterns/feature-flagged-rollout.md` for the stretch PDF/.docx parser. The docs contribution — `docs/patterns/brief-driven-generation.md` — is itself the promotion artefact for the runner contract + anchor protocol + review state machine; it ships as part of M12-6, not as an afterthought.
