# M12 — Brief-driven Sequential Page Generation

## What it is

A new generation mode that accepts a single whole-site brief (text or Markdown today; PDF/.docx stretch) and produces the full site one page at a time, each page generated with the entire brief in context plus a running summary of what preceding pages said. The operator uploads the document, a structural parser extracts an ordered per-page list with a per-page mode (`full_text` when the brief contains a complete section for that page, `short_brief` when it only sketches one), the operator reviews and commits the parse, and a sequential runner walks the list: draft → self-critique → revise → visual-review → revise again, pausing after each page for operator approval before advancing. DS-only generation is the default — no locked template per page, the runner composes from the site's design system and a shared `site_conventions` document that is fixed at page 1 and re-used verbatim for pages 2..N. Brand voice and design direction are first-class fields on the brief record, not hidden inside the document text, so the operator can edit them between pages without re-uploading. Pause-mode is the only review mode at launch; auto-advance is deferred until we've watched a dozen real sites go through the loop.

## Why a separate milestone

The existing batch generator (M3) and regeneration worker (M9) are per-page workers with no cross-page state — each slot is a leaf job, fan-out parallel, no ordering guarantees. Brief-driven generation inverts every one of those assumptions: it is sequential by contract (page 2 depends on page 1's committed conventions), it carries a growing content-summary context forward, it has a per-page review checkpoint the operator must clear, and it wants a multi-pass quality loop (self-critique + visual critique) that's too expensive to run on a 60-slot parallel batch. Folding this into M3 would either force the batch worker to learn a second execution shape or would quietly degrade batch throughput. M12 gets its own tables (`site_briefs`, `brief_pages`, `brief_runs`), its own runner (`lib/brief-runner.ts`), its own admin surface (`/admin/sites/[id]/briefs/[briefId]`), and reuses M7's publish path + M6's DS rendering without touching them.

The milestone is also write-safety-critical on a different axis than M3: a single brief run can spend 10-20× the Anthropic tokens of a batch slot (multi-pass × N pages × whole-doc context), so budget enforcement, idempotency, and resume-after-pause all have to land before the runner can touch a production tenant.

## Scope (shipped in M12)

- **M12-1 — schema + upload + structural parser + operator commit.** New tables `site_briefs`, `brief_pages`, `brief_runs` with the soft-delete + audit-column contract from `docs/DATA_CONVENTIONS.md`. Upload surface at `/admin/sites/[id]/briefs/new` accepts `text/plain` and `text/markdown`; file is stored in Supabase Storage under `site-briefs/<site_id>/<brief_id>.md`. Structural parser (`lib/brief-parser.ts`) prefers explicit structure first — Markdown H1/H2 sections, `---` separators, numbered "Page 1:" / "Page 2:" headers — and falls back to a single Claude inference call when no structure is detectable. Parser output is a draft `brief_pages` list with per-page `mode in ('full_text','short_brief')` based on whether the extracted section is long enough (>= 400 words) to be a complete page spec. Operator reviews the parse at `/admin/sites/[id]/briefs/[briefId]`, can re-order rows, edit titles, flip modes, add/remove pages, then commits — commit freezes the list and unlocks the runner.
- **M12-2 — brand voice + design direction + `site_conventions` schema + anchor-page spec.** `site_briefs` gains `brand_voice text` and `design_direction text` first-class fields populated from a form (pre-filled with Claude-inferred defaults the operator can override). New `site_conventions` JSONB column on `site_briefs` holds the exact design+content conventions the runner will reuse verbatim across every page: typographic scale choice, section rhythm, tone register, CTA phrasing, hero pattern, color-role mapping over the DS palette. The anchor-page spec (applied to page 1) runs 2-3 extra revision cycles beyond the standard multi-pass — the runner promotes the anchor's final `site_conventions` choices into the brief row, and pages 2..N receive them as hard constraints rather than suggestions.
- **M12-3 — sequential runner with multi-pass, whole-doc context, quality gates.** `lib/brief-runner.ts` processes one `brief_pages` row at a time. Per-page loop: draft (whole brief + `site_conventions` + running content summary + page spec) → self-critique (Claude re-reads its own draft against the brief and conventions, returns a list of gaps) → revise (applies the critique). Page 1 runs the loop 2-3 extra times to stabilise `site_conventions`. Output is staged in `brief_pages.draft_html` until the operator approves; approval promotes the draft to `generated_html` and appends a compressed 2-3 sentence summary to the brief's running `content_summary` that pages N+1.. will read. Quality gates reuse M7's runner contract (HTML size cap, DS-token whitelist, link integrity) — a gate failure blocks approval until the runner re-revises.
- **M12-4 — visual review pass.** After the multi-pass text loop lands a candidate, the runner spins up a Playwright worker to render `draft_html` against the site's DS, captures a viewport + full-page screenshot, and feeds both to Claude with the original page spec + `site_conventions`. Claude returns a structured critique (layout issues, visual hierarchy, contrast, white-space density, CTA prominence). If the critique flags any severity-high issue, the runner applies a revision pass that includes the screenshot feedback and re-renders. Capped at 2 visual iterations per page to bound cost. The final screenshot + critique log are stored against the `brief_pages` row so the operator sees exactly what Claude saw when approving.
- **M12-5 — review-between-pages UI (approve / cancel / resume).** `/admin/sites/[id]/briefs/[briefId]/run` is the runner surface. Shows the committed page list with per-row status (pending / generating / awaiting_review / approved / failed / cancelled), the current page's draft HTML preview + screenshot + Claude critique, and three action buttons: **approve** (advances the runner to the next page), **revise with note** (appends an operator note to the next revision pass), **cancel** (halts the run, leaves generated pages in place so the operator can keep or discard per page). The runner is pause-only: on every page completion it transitions to `awaiting_review` and waits for the operator action. Auto-advance is explicitly out of scope for M12.
- **M12-6 — E2E + docs + PDF/.docx parser (stretch).** Playwright spec covering upload → parse review → commit → run → approve page 1 → approve page 2 → cancel at page 3 → publish approved pages via the existing M7 surface. New `docs/patterns/brief-driven-generation.md` capturing the runner contract, the `site_conventions` anchor protocol, and the per-page review state machine. Stretch: if budget permits, add a PDF/.docx parser variant (`pdf-parse` + `mammoth`) gated behind a feature flag; if it slips, a BACKLOG entry carries it to a follow-up slice without blocking M12-5's merge.

## Out of scope (tracked in BACKLOG.md)

- **Auto-nav / menu generation.** The runner produces page HTML; it does not mutate the site's WP menu. Operator wires menus by hand after publish. Likely lands in a dedicated "site chrome" slice that also covers footer, header nav, breadcrumb generation — too large to bolt onto M12.
- **`sitemap.xml` generation.** Same reasoning. WordPress plugins already produce sitemaps on publish; we do not want to own that surface until we have a reason to.
- **Multi-language briefs.** One brief = one language today. A multi-language brief would need per-language `site_conventions`, per-language running summaries, and per-language review surfaces — a 2-3× scope multiplier. Belongs to an i18n milestone.
- **Brief versioning + diff.** Today an edited brief is a new brief row (soft-delete the old one). Full version history + per-field diff + "re-run page 4 with v2 of the brief" is a separate capability that would need its own data model. Not needed for the first dozen real sites.
- **Auto-advance review mode.** Pause-only for launch. Auto-advance is a one-flag change once we trust the visual-review pass — but "trust" here means watching enough real runs that we know what a bad visual critique looks like. Deferred explicitly.

## Env vars required

None new beyond existing `ANTHROPIC_API_KEY`, the Supabase trio (`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`), and the already-provisioned Langfuse + Playwright setup M7/M11 rely on. The visual-review pass reuses the existing Playwright worker infrastructure; no new screenshot service, no new storage bucket beyond the existing `site-briefs/` prefix added in M12-1.
