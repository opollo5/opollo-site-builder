# Discovered Issues ‚Äî Phase 1 Inventory

Issues found during codebase investigation. These are NOT fixed here ‚Äî each
item is logged with a citation and a suggested follow-up. Actual fixes go in
separate PRs.

Severity: **P0** = data corruption / security / always-broken; **P1** = wrong
behaviour visible to user; **P2** = polish / missing coverage.

---

## DI-001 ‚Äî `user` role declared but never granted or enforced ‚Äî P2

**Found during:** roles-and-permissions.md investigation
**File:** `lib/auth.ts:16` ‚Äî `export type Role = "super_admin" | "admin" | "user"`
**Detail:** The `user` role appears in the TypeScript type and the DB CHECK
constraint but no route grants it and no route requires it. `checkAdminAccess()`
only allows `super_admin` and `admin`. An `opollo_users` row with `role='user'`
would be redirected to `/` from every admin page with no explanation.
**Suggested action:** Either remove from the type and DB enum, or document what
surfaces it should access.

## DI-002 ‚Äî `/company/social/connections/connect/[platform]/page.tsx` is a stub redirect ‚Äî P1

**Found during:** routes-and-pages.md investigation
**File:** `app/(platform)/company/social/connections/connect/[platform]/page.tsx:7`

## DI-003 ‚Äî Bulk CSV upload does not verify `schedule_post` permission ‚Äî P0

**Found during:** state-machines.md / roles-and-permissions.md investigation
**File:** `app/api/platform/social/drafts/bulk/route.ts`
**Detail:** The bulk CSV upload endpoint sets `state='scheduled'` based on CSV
data without verifying the caller has the `schedule_post` permission
(`approver+` required). An `editor`-role user who uploads a CSV with a
`scheduled_at` column can bypass the permission gate.
**Related:** G8 in `docs/investigations/composer-state-gaps-backlog-2026-05-25.md`
**Suggested action:** Add `canDo("schedule_post", session)` check before
setting state='scheduled' in bulk rows, or force all bulk imports to
`state='draft'` and require an explicit schedule action.

## DI-004 ‚Äî Missing `loading.tsx` on 97 of 100 page routes ‚Äî P2

**Found during:** routes-and-pages.md investigation
**Detail:** Only 2 pages have colocated `loading.tsx` files:
- `app/(platform)/company/social/analytics/loading.tsx`
- `app/(platform)/social/poster/loading.tsx`

All other pages have no loading boundary. Next.js App Router falls back to
the nearest ancestor Suspense boundary when no colocated `loading.tsx` is
present, which in most cases is the root layout ‚Äî meaning the entire page
shell disappears during navigation.
**Suggested action:** Audit the highest-traffic routes (calendar, posts list,
admin sites, admin companies) and add colocated `loading.tsx` stubs showing
skeleton placeholders to prevent layout flash.

## DI-006 ‚Äî `social_post_master` (V1) and `social_post_drafts` (V2) coexist ‚Äî P1

**Found during:** state-machines.md investigation
**Files:**
- `lib/social/types.ts` ‚Äî V2 types (social_post_drafts)
- `lib/platform/social/posts.ts` ‚Äî V1 types (social_post_master)
**Detail:** Two separate post data models exist:
- V1 (`social_post_master` / `social_post_variant`) ‚Äî used by the older
  `/company/social/posts/[id]` tabbed post detail view
- V2 (`social_post_drafts`) ‚Äî used by the composer and calendar

The two models have different state enums. V1 uses states like
`pending_client_approval`, `changes_requested`, `pending_msp_release`
that do not exist in V2's `lib/social/post-state-actions.ts`.
This creates UI surfaces where the same post state has different names
and different allowed actions depending on which view renders it.
**Suggested action:** Clarify migration path from V1 to V2 and
ensure the state matrices are synchronized or the two surfaces are
clearly separated.

## DI-007 ‚Äî Implicit Opollo staff admin grant is not audit-logged ‚Äî P1

**Found during:** roles-and-permissions.md investigation
**File:** `lib/platform/auth/current-user.ts:132`
**Detail:** When Opollo staff navigate to a customer company without a
`platform_company_users` row, the system silently grants them an `admin`
role. This means staff can make mutations (create posts, edit drafts, manage
connections) as a company admin without the customer knowing or any log entry.
**Suggested action:** Consider adding a `service_access_log` entry or
equivalent when staff access company data, or restrict implicit staff grant
to read-only for data mutation endpoints.

## DI-009 ‚Äî Review token route uses JWT but token revocation is not documented ‚Äî P1

**Found during:** routes-and-pages.md investigation
**File:** `app/(public)/review/[token]/page.tsx`
**Detail:** The review link uses a JWT with 14-day expiry. Once issued, a
review link cannot be revoked before expiry unless the post itself is deleted.
If a post is shared for review and then rejected/cancelled, the review link
may still be accessible.
**Suggested action:** Clarify whether `review_link` rows are invalidated when
a post transitions out of `pending_approval` state, and document the behaviour.

## DI-010 ‚Äî Approval toggle + `schedule_post` permission gap ‚Äî P1

**Found during:** components-catalog.md + roles-and-permissions.md investigation
**File:** `components/social/composer/ApprovalToggle.tsx`
**Detail:** The `ApprovalToggle` component lets users toggle whether a post
requires approval before publishing. If an editor (who lacks `schedule_post`
permission) can toggle `approval_required=false`, they might be able to
schedule a post directly without approval ‚Äî bypassing the approver gate.
The server-side enforcement of this toggle needs verification.
**Suggested action:** Verify that toggling `approval_required=false` is
gated at the API level to `approver+` role, not just in the UI.
